mit6.s081-lab2

实验背景

lab2 的主要内容是为 xv6 增加两个系统调用 trace 和 sysinfotest。在做实验之前先阅读 xv6 book 的第二章和第四章的几节内容。比较有趣的内容是对 xv6 启动流程的介绍：

  • RISC-V 机器通电后，存储在 ROM 中的 boot loader 程序把 xv6 内核加载到内存 0x80000000（此时虚拟地址尚未启用，此为物理地址。不用 0x0 是因为0x0:0x80000000 范围是 I/O 设备），CPU 会在机器模式下从 kernel/entry.S 的 _entry 入口开始执行指令；
  • _entry 会设定一个初始栈（定义在 kernel/start.c 中），把 sp 寄存器的值设为此栈末尾（RISC-V 的栈向下增长）并调用 kernel/start.c 的 start 函数；
  • start 函数进行一些配置工作，启动时钟并从机器模式返回特权模式，然后调用 kernel/main.c 中的 main 函数；
  • main 会初始化一些设备和子系统，接下来会调用 kernel/proc.c 的 userinit 函数以创建第一个进程。
  • 第一个进程通过 exec 系统调用把自己替换为运行 user/init.c 的 init 进程，init 则会创建控制台设备文件，切换输入输出文件，并在控制台上创建一个 shell。

xv6 系统调用实现

本实验不需要了解实现系统调用的低层细节（在实验 4 Traps 会涉及），但系统调用的实现机制和调用库函数相去甚远，所以需要了解系统调用的大致架构。

以题目一 trace 为例，如果要实现完整的 trace，那么要达到以下要求：

  1. 题目给出了用户态的 user/trace.c 文件，该文件调用 trace 函数；

  3. usys.S 设置 a7 寄存器为系统调用号，用 ecall 指令进行系统调用；
  4. 中断处理过程，后续会具体研究；
  5. 内核通过 kernel/syscall.c 中的 syscall 函数，根据参数找到所需系统调用并执行。

实验题目

trace

本实验第一个题目是为 xv6 增加一个 trace 系统调用，进行 trace 系统调用后的进程所进行的每个系统调用，都会被内核检查：如果该系统调用的编号在 trace 的参数中，内核就会打印这个系统调用的名称和返回值，该进程通过 fork 产生的所有子进程也都会被检查。trace 的参数是用整型作为二进制掩码传递的，例如 read 的编号为 5，如果要跟踪该进程的 read 系统调用，就要让参数的第五个二进制位为 1。

为了实现上述功能，首先要在 user/user.h user/usys.pl kernel/syscall.h 中加入 trace 的相应内容，然后在 kernel/sysproc.c 中新增一个 sys_trace 函数来实现 trace 的功能。

考虑如何实现 trace。考虑到只有调用 trace 的进程的系统调用会被内核打印，而且子进程会遗传这个特性，因此可以在 kernel/proc.h 的进程结构体中加入一个 trace_mask 属性，然后让 sys_trace 通过 myproc 设定当前进程的 trace_mask；与此同时，在 kernel/syscall.c 的 syscall 函数中，在系统调用返回前根据 trace_mask 输出系统调用即可。记得要更新 kernel/proc.c 中的 fork 函数，使 trace_mask 在父子进程间传递。
